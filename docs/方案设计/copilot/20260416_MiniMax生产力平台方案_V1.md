概述
本方案为一套可落地的生产力平台设计，目标是把 MiniMax 的能力（文本生成、文生图、图生图、embeddings、RAG 等）做成一个正规、可扩展的产品级平台，并在设计上考虑用户隔离、权限与后续扩展（视频、语音、批量任务）。文档按方案组分组，便于后续选择不同实现路线。

分组设计（按优先级）

组 A：稳健可扩展平台（推荐）
- 目标：面向生产的分层架构，支持文本、图像、embeddings、RAG、多租户隔离，后续可接入视频/语音。
- 核心模块：接口层（`api/`）、服务封装（`service/minimax_service.py`）、数据模型（`schemas/`）、前端（`templates/`、`static/`）、存储/向量库（建议新增 `db/`、`vector_store/`）、异步队列（建议新增 `worker/`）。
- 特点：安全（密钥后端管理）、可观测（日志/监控）、可扩展（抽象客户端）。

组 B：轻量验证方案（PoC）
- 目标：最小改动快速验证核心业务（短剧文本、基础文生图），优先验证用户体验与价值。
- 特点：低成本、快速交付，后续需补齐安全与扩展能力。

组 C：全能创作扩展（可选，高成本）
- 目标：接入所有 MiniMax 能力（含视频、TTS/ASR、批量），并支持管理后台、配额计费。
- 风险：基础设施与运营成本高。

关键非功能需求（通用）
- 安全：API Key 仅后端环境变量，前端不存敏感信息；所有数据按 `owner_id` 进行隔离。
- 权限：支持 `admin` 与 `user`，后续可扩展为组织/租户。
- 可测性：将外部 HTTP 客户端抽象以便 mock，服务层写单元测试。
- 可观测：记录请求日志、用量统计、错误上报（建议 Sentry）。

短期实现建议（路线）
1. 抽象并实现 `minimax_service.py` 的 text-post 封装，增加统一错误处理与重试策略。
2. 前端支持从 `static/config/model_config.json` 读取配置并覆盖请求体；前端不包含任何密钥。
3. 新增用户注册/登录与基础 RBAC（JWT），实现数据隔离。
4. 实现 embeddings 生成与简单检索（RAG demo）。
5. 实现文生图接口并通过异步任务队列处理生成（任务状态 API）。

开发注意点与风险
- 不在前端暴露 API Key；"配置文件" 只能放非敏感项。
- 控制输入大小与频率，避免高额消耗。
- Embeddings 与文件存储要考虑隐私与成本。
